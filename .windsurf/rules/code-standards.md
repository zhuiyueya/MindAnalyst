涉及后端代码修改时必须遵守：

先读再改：改动前先定位相关文件/调用链，确认真实行为与约束。
最小改动：只改与需求直接相关的内容；避免无意义重构、重命名、格式化、搬文件。
分层清晰：入口层只做入参/出参与调用；业务编排在 service；数据访问集中到 repository；外部依赖集中到 adapter。
依赖单向：高层调用低层，禁止低层反向依赖高层。
类型尽量明确：公共接口/返回结构优先强类型；Any 只用于外部不可信输入/输出边界，进入核心逻辑前要收敛。
统一错误策略：失败要么明确抛错，要么明确降级；不要静默吞错；关键路径加日志。
配置集中：可变参数（模型名、URL、key、阈值等）必须走配置/setting，不硬编码在业务逻辑里。
避免重复初始化：重资源（模型/客户端/连接）应复用或集中创建，防止在热路径里反复 new。
保持兼容：除非明确要求，外部 API/数据结构不破坏；需要变更时提供迁移方案。
改完要可运行：确保导入不破、基本自检通过（至少能编译/启动关键链路）。
Adapter 边界强类型：adapter 层对外输入/输出必须用 Pydantic DTO 严格校验并收敛 Any/dict，进入核心逻辑后只处理结构化数据，失败要显式返回错误或抛错。
提示词统一管理：所有系统提示词/模板/场景 prompt 必须通过项目的提示词管理模块统一维护（src/prompts/PromptManager + PromptRegistry + templates/profiles.yaml 体系）；禁止在业务代码中散落硬编码长 prompt 文本（允许保留极短的 fallback/错误提示）。
不确定就问：需求/边界不清时先提出具体澄清问题再动手。
